### 关键纪律（避免 5 人协作踩坑）

1. **依赖方向硬约束**（03 §4.2）：反向依赖即 CI 失败。`config` 不得 import `memory`，`tools` 不得 import `api`。架构测试由丁 M1 写好，全员遵守。
2. **接口契约先行**（03 §4.3）：Protocol 优先。M1 阶段甲必须先定 `BaseAgent` / `BaseChatModel` / `MemoryBackend` 三个核心 Protocol，其他人在 Protocol 上写实现。
3. **每个里程碑有可运行产物**：M1 → 单图；M2 → 双层记忆；M3 → 三 Agent Sequential；M4 → E2E。中间状态都要 `pytest tests/architecture` 通过。
4. **并行纪律**：M2 期间丁的 LSP 客户端需要 pyright 子进程，先在本地跑通；M3 期间乙的 MCP 注册表需要等甲的 multi_agent 接口定稿才能用——避免接口定义晚于实现。
5. **共享中间件**（02 §9）：戊的 `middleware` 是横切模块，所有人 M3 都要接入。所以戊在 M2 末就要提前出一个 `MiddlewareChain` 骨架。

### 风险与备选

- **丁的 LSP 客户端风险最高**（pyright 子进程跨平台 + Python LSP 协议复杂）。如果 M2 跑不通，可以降级为"仅 AST"，LSP 标 Could 留到 M4。
- **乙的 MCP 切换风险**（自研 vs `langchain_mcp_adapters`）。M1 用自研，M4 切换前必须做一遍对比测试，否则生产环境有兼容坑。
- **丙的 ChromaDB 持久化风险**（向量库在 Windows 上的性能）。M2 末要先压一遍 10k doc 写入 + 检索。
